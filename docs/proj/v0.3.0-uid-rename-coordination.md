# v0.3.0 — `uid` 全仓重命名为 `id` / `user_id` 实际影响修正版

> 📝 **2026-05-22 修正**：本文最初版本夸大了"硬切 / token 全失效 / 老 URL 404"的紧迫感。
> 经过三仓代码扫描，**实际跨仓 wire impact 为 0**：
>
> 1. **JWT claim `uid → id`**：kungal Go 0 处解 wiki JWT、moyu Go 0 处解 wiki JWT。两家都是 pass-through 模式（拿 OAuth access_token 转发给 wiki，自己不验签），走 `/userinfo` HTTP body 拿用户信息（响应字段一直是 `"id"`，与 JWT claim 名无关）。
> 2. **URL 路由参数 `:uid → :id`**：HTTP request URL **不含参数名**。客户端发的是 `/contributors/42`，server 怎么注册路径参数名不影响匹配。kungal/moyu 早就用 `/contributors/:id` 注册自己的透传路由。
> 3. **Fiber context key `user_uid → user_id`**：纯本仓内部 middleware↔handler 协议。
>
> **本仓 v0.3.0 本质是命名一致性 PR，不是 wire breaking PR**。kungal/moyu 不需要为本仓改动做任何"同步部署"动作；他们各自的内部命名清理是独立的工作。

---

## 1. 本仓（kun-galgame-infra）已经改了什么

### 1.1 JWT claim 重命名

```go
// apps/api/pkg/utils/jwt.go
type TokenClaims struct {
    UserUUID string `json:"sub"`
    ID       uint   `json:"id"`    // ← 旧：UID `json:"uid"`
    // ...
}
```

**影响**：access_token 的 JSON payload 字段 `uid` 改名为 `id`。新 token 长这样：

```json
{
  "sub": "1234567890abcdef-...-uuid",
  "id": 42,
  "email": "user@example.com",
  ...
}
```

### 1.2 Fiber context key 重命名

```go
// 旧：c.Locals("user_uid")  →  新：c.Locals("user_id")
```

仅本仓内部 middleware ↔ handler 协议，**不跨仓**。kungal/moyu 不需要管。

### 1.3 URL 路由参数重命名

```
/api/v1/galgame/user/:uid/stats          →  /api/v1/galgame/user/:id/stats
/api/v1/galgame/:gid/contributors/:uid   →  /api/v1/galgame/:gid/contributors/:id
```

**所有 kungal/moyu 透传到 wiki 这两个 endpoint 的代码必须更新 URL 模板**。

### 1.4 nitro-server `userInfo.uid` → `userInfo.id`

本仓的 `apps/nitro-server` 是 dead code（无 package.json），改动只为 grep 清洁。kungal/moyu 自己的 nitro-server 是活跃代码，需要单独处理（见 §2.3）。

### 1.5 内部变量名

Go：函数参数和局部 `uid int` → `userID int`（约 80 处，纯内部重命名，零 wire 影响）
TS：nitro-server 内部 `const uid = userInfo.id` → `const userId = userInfo.id`（dead code 清理）

## 2. kungal / moyu 该做什么

### 2.1 JWT verifier 必须改

每个仓里读 JWT claim 的地方都要把 `claims.UID` / `claims["uid"]` 改成 `claims.ID` / `claims["id"]`。

**Go 端**（kungal-api / moyu-api）：

```bash
# 在每个 Go 仓里 grep
grep -rn 'claims\.UID\|json:"uid"' apps/api --include='*.go'

# 应用同款修改
# 1. TokenClaims 结构体字段 UID → ID，JSON tag "uid" → "id"
# 2. 所有 claims.UID 读取点 → claims.ID
```

**TS 端**（nitro-server 验证 JWT 的地方）：

```bash
# 找 nitro-server 验证 JWT 的 utility
grep -rn 'decode.*token\|verify.*token\|claims\.uid\|claims\["uid"\]' apps/nitro-server --include='*.ts'

# 把 claims.uid 改成 claims.id
```

### 2.2 URL 透传层更新

kungal/moyu 的 Go API 如果做了 wiki 透传（比如 `/galgame/user/:uid/stats` 代理到 wiki），把透传规则的 URL 模板从 `:uid` 改成 `:id`。

**关键文件**（基于本仓 §13 同步审计时观察）：
- kungal: `apps/api/internal/galgame/client/path_mapper.go`
- moyu: 等价位置

### 2.3 nitro-server `userInfo.uid` → `userInfo.id`

两个仓的 nitro-server 都活跃。需要：

1. **类型层**：`UserInfo` interface `uid: number` → `id: number`
2. **session adapter**：读 JWT claim `id`（不是 `uid`）
3. **所有调用点**：`userInfo.uid` / `userInfo?.uid` → `userInfo.id` / `userInfo?.id`

sed 命令：

```bash
# nitro-server 内部
find apps/nitro-server -name '*.ts' | xargs sed -i \
  -e 's/userInfo\.uid/userInfo.id/g' \
  -e 's/userInfo?\.uid/userInfo?.id/g'

# 类型定义文件（手动改 interface UserInfo { uid → id }）
grep -rn 'interface UserInfo\|type UserInfo\b' apps
```

### 2.4 内部变量名重命名（可选但建议）

Go 端把函数参数 `uid int` 改成 `userID int`：

```bash
# 找出所有用 uid 当 user_id 语义的 Go 文件（避开 Meilisearch index uid）
find apps/api -name '*.go' \
  | grep -v 'search/' \
  | xargs grep -l '\buid\b'

# 然后 sed 同款（用 Phase 2 的命令）：
xargs sed -i 's/\buid\b/userID/g'
xargs sed -i 's/\bviewerUID\b/viewerUserID/g; s/\btargetUID\b/targetUserID/g'
```

TS 端 nitro：

```bash
# const uid = userInfo.id  →  const userId = userInfo.id
grep -rln 'const uid = userInfo' apps/nitro-server --include='*.ts' \
  | xargs sed -i 's/const uid = userInfo/const userId = userInfo/g; s/\buid\b/userId/g'
```

### 2.5 URL 路由参数（如果 kungal/moyu 有自己的 `/user/:uid/...` 路由）

```bash
# Go fiber
grep -rn ':uid' apps/api --include='*.go'
# nitro file-based
find apps/nitro-server -path '*\[uid\]*'
# Vue page routes
find apps/web/app/pages -path '*\[uid\]*'
```

凡是用户语义的 `:uid` / `[uid]`，全部改 `:id` / `[id]`。

## 3. 不该改的（Category C，本仓**保留**）

| 位置 | 为什么不改 |
|---|---|
| **Meilisearch index `uid`**（`apps/api/internal/infrastructure/search/client.go`、`apps/api/internal/platform/galgame/search/settings.go`） | `uid` 在 Meilisearch 官方 API 里就是"索引唯一标识"的术语，跟用户无关。改了反而引起新误解 |
| **`UUID` 字段**（`User.UUID`, `c.Params("uuid")` 这种字符串 UUID） | 这是 UUID（universally unique identifier）字符串，不是 user_id 整数 |
| **`useId` / `kunUniqueId` Vue composable** | Vue 内部生成的 HTML id，跟用户无关 |
| **`useFocusTrap`** | a11y 库 API，名字撞字 |

## 4. 部署方案（修正版 —— 不需要协同窗口）

**本仓 v0.3.0 可以独立部署，零协调成本**。理由见文档顶部修正说明。

### 本仓
- 任意时机重启 wiki API、web、wiki 三个 app 即可
- 用户**不需要重新登录**（JWT 自验自用，wiki 启动后新签的 token claim 已是 `id`，老 token 在 max-age 内仍可解 —— 但实际本仓 access_token 默认 1h、refresh_token 30d，过期周期自然轮转）

### kungal / moyu
- 内部 `uid` 命名清理（§2）是他们各自独立的工作
- 跨仓 wire 不受影响 —— **他们可以等任意时机做、不做也不影响 wiki 服务**

### 部署后验证（本仓）
- 新签 token 解码后 payload 含 `"id"` 字段（不再有 `"uid"`）
- `/api/v1/galgame/user/:id/stats` endpoint 接受任何整数 ID 参数

