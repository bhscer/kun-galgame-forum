# v0.5.2 — KunDrawer 响应式行为（加 `responsive`）

## 响应式行为（v0.5.2 加 `responsive`）

> **默认开箱即用**：桌面端从右侧滑入，手机端（viewport < `md` = 768px）自动改成从底部滑入。这是绝大多数 drawer 用法的合理默认 —— 移动屏太窄塞不下右侧抽屉。

```vue
<!-- 默认行为：桌面 right，手机 bottom -->
<KunDrawer v-model="open" />

<!-- 显式锁死 placement，所有 viewport 都从顶部滑入 -->
<KunDrawer v-model="open" placement="top" :responsive="false" />

<!-- 桌面左侧 + 手机底部 -->
<KunDrawer v-model="open" placement="left" />
```

实现细节：

```ts
import { useMediaQuery } from '@vueuse/core'

// 匹配 Tailwind 的 md breakpoint（默认 48rem = 768px）
const isMobile = useMediaQuery('(max-width: 47.99rem)')

const effectivePlacement = computed<KunDrawerPlacement>(() =>
  props.responsive && isMobile.value ? 'bottom' : props.placement
)
```

`effectivePlacement` 同时控制 `panelGeometry`、`transitionName`、`rounded` 圆角侧选择 —— **手机断点跨越时**自动重算几何 + 切换 slide 方向。

### SSR 安全性

`useMediaQuery` 在 SSR 期返回 `false`，client 首帧 resolve 真实值。**因为 KunDrawer 默认 `v-model` 是 `false`（关闭），drawer 子树根本不会进入 SSR HTML**，所以不存在"桌面 vs 手机不同 placement 导致 hydration mismatch"的问题。Drawer 实际挂载是用户点开后，那时 client JS 早已 alive 且 isMobile 已 resolve。✓

### 已知边角

打开 drawer **同时**用户跨越 768px 断点（极少：拖窗口宽度 / 平板旋转横竖屏）时，`panelGeometry` reactive 变化 → panel 的 class 改 → **会"跳"到新位置而不是滑过去**（Vue Transition 只在 `v-if` 切换时触发）。这种 edge case 不修，因为：
- 频率极低：用户极少在打开 drawer 时拖窗口
- 修复成本高：要监听 isMobile 切换，关闭再重开，焦点丢失
- 视觉上"跳"也比"卡在错误位置"好
