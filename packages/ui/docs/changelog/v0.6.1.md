# v0.6.1 — KunContent 内置图片 lightbox（评论/正文图片可点开）

承接 v0.6.0 的 KunLightbox 重写。本版把"渲染后的 markdown 图片点击放大"能力**沉进 `KunContent` 组件内部** —— 任何 `<KunContent :content>` 的内联图片自动可点开 lightbox，消费方零样板。

## 背景：为什么放 KunContent 而不是独立 composable

下游论坛（moyu / kungal）的评论、回复、正文都是后端 markdown 渲染成 `*_html` → 前端 `v-html` 输出**裸 `<img>` DOM**。需求：点这些图片弹 lightbox，且翻页/异步加载后新增的图片也要能点。

两个约束决定了方案：

1. **不能用 `<KunLightboxGalleryItem>`** —— 那需要每张图是 Vue component，但 v-html 出来的是裸 DOM。
2. **不能用 onMounted 一次性扫描 + 给每个 img 绑 listener** —— 翻页后新增的 img 没有 listener，点了没反应。

→ **事件委托**：listener 绑在持久的 prose 容器上，点击时**实时 `querySelectorAll('img')`**，不缓存。翻页 replace / append / 异步加载完都天然支持，零重新初始化。

为什么沉进 KunContent 而非做成消费方手动接的 composable：

- `KunContent` 已经是 markdown prose 的**统一渲染入口**（DOMPurify sanitize + `kun-prose` class + `useSpoilerContent` 注入 spoiler 交互）
- 图片 lightbox 与 spoiler 是**完全同形状**的需求（给 v-html 输出注入 DOM 级交互），应该并列坐在 KunContent 里
- 消费方 `<KunContent :content>` 本来就这么用 → 图片自动可点，**不可能漏接**（对比手动 composable：每处要 destructure 4 ref + `@click` + 放 `<KunLightbox>`，9 处漏一处就没效果）

## 改动

### 新增 `useContentLightbox(containerRef)`

`packages/ui/app/composables/topic/useContentLightbox.ts`。跟 `useSpoilerContent` 并列（同 `topic/` 目录、同 `(containerRef)` 签名）。

```ts
const { isLightboxOpen, images, currentImageIndex } = useContentLightbox(articleRef)
```

- 绑 `click` 到 containerRef 元素（innerHTML 变化不替换该元素，listener 存活）
- 点击 `<img>` → 实时扫描容器内全部 img → 找 index → 开 lightbox
- `e.stopPropagation()` + `e.preventDefault()` —— 点图片只开 lightbox，不触发祖先 click（如评论卡片导航）
- gallery 作用域 = 该容器自己的图片集（点 A 评论的图，←/→ 只在 A 评论内切换）

### KunContent 内置

`packages/ui/app/components/kun/content/Content.vue`：

```vue
const { isLightboxOpen, images, currentImageIndex } = useContentLightbox(articleRef)
```
```vue
<KunLightbox v-model:is-open="isLightboxOpen" :images :initial-index="currentImageIndex" />
```

+ `.kun-prose :deep(img) { cursor: zoom-in }` 提示可点。

### 删除老 `useKunLightbox`

`packages/ui/app/composables/useKunLightbox.ts` 删除。它是 dead code（canonical 零真实调用方）+ 实现有缺陷：
- `onMounted` 一次性扫描 → 翻页新图绑不上
- 依赖 `img[data-kun-lazy-image]` 属性 → 后端 markdown 渲染根本不带这属性
- 给每个 img 单独 addEventListener → 无容器隔离

被 `useContentLightbox` 完整取代。**这是上游 drift 收口**：下游 moyu fork 之前自行重写过本地副本（事件委托版），现在 canonical 出了正式版本，下游应删本地副本回归跟随 upstream（见 handoff）。

## 与现有体系的一致性

| 关注点 | 处理 |
|---|---|
| 与 spoiler 交互 | 并列：spoiler 先 reveal，再点图开 lightbox。`.kun-prose img` cursor 不影响 spoiler 内的图（spoiler 自己有 cursor:pointer 覆盖）|
| sanitize | 不变，KunContent 仍 DOMPurify。lightbox 只读已 sanitize 后 DOM 的 `<img src>` |
| KunLightbox 能力 | 全继承 v0.6.0（滑动切图 / 旋转 / 缩放 / 工具条 / dark mode / 下载 fallback）|
| 翻页 / 异步 | 实时扫描天然支持 |

## 影响面

- galgame-infra web / wiki：几乎不渲染 markdown 评论，`<KunContent>` 当前无调用方 —— 本版对它们是 latent 能力（将来用到即生效）
- moyu / kungal：真正受益方。需把手搓 `<div class="kun-prose" v-html>` 迁移到 `<KunContent :content>` 才白嫖（见 handoff 迁移指令）。在迁移前，可临时直接调 `useContentLightbox(ownRef)` + 手放 `<KunLightbox>` 作为 escape-hatch。
